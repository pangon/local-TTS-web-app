# Backend

**Responsibility**: HTTP request handling, REST API, SSE endpoint, application services (Library, Job, Model, Monitor), SQLite storage, static file serving, and all TTS inference and GPU interaction via a modular subpackage.

**Technology**: Python, FastAPI, Uvicorn, SQLite (via `sqlite3` stdlib), PyTorch (CUDA), HuggingFace Transformers, huggingface_hub

## Interfaces

- HTTP REST + SSE with frontend: serves API endpoints and pushes real-time events
- TTS subpackage (internal): application services import and call `TTSEngine` for GPU inference, model management, and chapter parsing (`DEC-tts-as-backend-module`)
- Model adapter layer (internal): per-model loading and inference is delegated to concrete adapters implementing the `ModelAdapter` protocol (`src/local_tts/tts/adapters/`). New models are enabled by adding an adapter and registering it in `_ADAPTER_REGISTRY`; `ModelLoader.list_models()` exposes a `loader_available` flag (surfaced through `GET /models`) so the frontend can hide download/load actions for models without an adapter. **Language convention:** the application layer passes the canonical **ISO 639-1 code** (`it`, `en`, … — same as preprocessing / `DEC-default-italian-language`) to `synthesize(language=…)`; each adapter must translate that code to its model-specific identifier (Kokoro `it`→`i` via `_resolve_lang_code`; Qwen3 `it`→`Italian` via `_resolve_language`), still accept its native identifiers for backward compatibility, and raise a clear `ValueError` for an unrecognized language. See `architecture.md` § Model-Specific Loading Requirements.
- Preprocessing service (internal, `src/local_tts/preprocessing/`): a dedicated application service — **not** in the TTS subpackage (`DEC-text-preprocessing-pipeline`). A `Pipeline` runs an ordered list of discrete `Stage` objects (the `Stage` protocol + a name-keyed registry in `stages.py`), configured along two axes — language profile + model profile (`profiles.py`) — with an optional on-disk domain dictionary tolerant of absence. `PreprocessingService.preprocess(text, language, model_id)` returns normalized text + before/after char counts. New stages are added by implementing the `Stage` protocol under the canonical name constants (`STAGE_*`) and registering them via `register_stage` in `preprocessing/__init__.py`; the default model profile runs the registered subset of `DEFAULT_STAGE_ORDER`. Stage convention (established by `unicode_sanitization.py`, the first concrete stage): one module per stage at `preprocessing/<stage_name>.py`; universal tables live in the module while language-specific data is exposed as a module-level `BUILTIN_LANGUAGE_DATA` mapping (`{language: {namespace: {...}}}`, the namespace named after the stage) and registered via `register_language_data` next to `register_stage` in `__init__.py` (a stage with no language-specific data omits `BUILTIN_LANGUAGE_DATA` and just registers the stage, as `layout_repair.py` and `sentence_segmentation.py` do); behavior switches read from the model profile's `params` via `PARAM_*` constants with safe defaults — no language/model rule is hardcoded in the shared `run` logic (`DEC-text-preprocessing-pipeline`). The default order is: unicode sanitization → layout repair → numeric/symbolic verbalization → abbreviation expansion → **sentence segmentation**. `sentence_segmentation.py` (the fifth stage) puts each sentence on its own line; the synthesizer chunks the confirmed text **one line per segment** (`split_into_sentences` in `tts/synthesizer.py` splits on newlines only, not punctuation), so this stage *defines* the TTS chunking and the reviewed text matches the spoken chunks. It runs **last** because splitting on a sentence-ending period must happen after numbers/abbreviations are expanded (otherwise `11.988` / `E.F.` are mis-split) — the inverse of layout repair, which *reflows* soft-wrapped fragments and keeps heading/list/**bare-number** lines standalone so structural lines are not glued into prose. It also performs **dialogue isolation**: a spoken span delimited by directional guillemets (`«` … `»`) is put on its own line(s) by breaking before `«` and after `»` (so narration / quote / trailing dialogue tag are distinct chunks), then flattens `«`/`»` to `"`. For this to work the **Unicode stage deliberately does NOT flatten `«`/`»`** (it flattens every other quote variant) — the directional guillemets carry chunking information consumed downstream; any new quote-related handling must preserve that contract. A stage may delegate language-specific work to a declared library selected by the language profile rather than branch on the language itself: `numeric_symbolic_verbalization.py` (the third concrete stage) spells numbers/ordinals/years via `num2words` (LGPL, a direct dependency) keyed by a `num2words_lang` code in its `BUILTIN_LANGUAGE_DATA`, with every other table (separators, month names, currency/percent/symbol words) in that same language data. A requested output language with **no** registered data is rejected by `resolve_language_profile` with `UnsupportedLanguageError` (surfaced by `/preprocess` as a 400) rather than degrading to a no-op — preprocessing rewrites are language-specific, so a passthrough would misrepresent the raw text as "normalized"; an empty/omitted language falls back to the supported default (`it`). Individual stages remain defensively no-op when the resolved (supported) language data lacks their own namespace. The `model_id` of the loaded model is supplied by the caller (the `/preprocess` API layer reads it from the Model Service and owns the "no model loaded" error), keeping the service unit-testable without a GPU.
- Preprocess & synthesis API (`src/local_tts/api/preprocess.py`, `jobs.py`): the synchronous `POST /preprocess` accepts a multipart `.txt` **file** _or_ raw `text` plus an optional `language`, runs the Preprocessing Service with the **currently loaded model's** profile, and returns the normalized text with before/after char counts for review (`REQ-USA-normalized-text-review`). The API layer owns input validation (`.txt`/≤2 MB/UTF-8/non-empty, exactly-one-of file/text, and an unsupported output `language` → 400) and the "no model loaded" error (409), reading `tts_engine.loaded_model_id` so the service stays GPU-agnostic; `REQ-F-upload-text-file` is satisfied here. Language validation precedes the model check (input-validation-before-model-check), so an unsupported language returns 400 even with no model loaded. `POST /jobs/synthesis` then takes JSON `{text, source_filename, voice?, language?}` and synthesizes **exactly** that confirmed text with **no** re-preprocessing (disk preflight estimated from `len(text)`). This two-call, transient-normalized-text contract is the preprocess-then-confirm flow (`DEC-preprocess-review-flow`). The `PreprocessingService` is instantiated once in the app lifespan and attached to `app.state`. All routes mount under `/api/v1`.
- Library & playback API (`src/local_tts/api/audiobooks.py`, `playback.py`): the `LibraryService` backs audiobook list/get/delete (delete cascades to chapter records and on-disk audio) and resolves chapter audio paths; chapter audio is streamed inline via `FileResponse` with HTTP Range support (206/`Content-Range`/416). Playback resume is a two-level bookmark (audiobook-level last chapter + per-chapter timestamp) served by the dedicated `PlaybackService` under `GET`/`PUT /audiobooks/{id}/position`.
