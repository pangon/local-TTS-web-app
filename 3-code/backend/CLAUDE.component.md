# Backend

**Responsibility**: HTTP request handling, REST API, SSE endpoint, application services (Library, Job, Model, Monitor), SQLite storage, static file serving, and all TTS inference and GPU interaction via a modular subpackage.

**Technology**: Python, FastAPI, Uvicorn, SQLite (via `sqlite3` stdlib), PyTorch (CUDA), HuggingFace Transformers, huggingface_hub

## Interfaces

- HTTP REST + SSE with frontend: serves API endpoints and pushes real-time events
- TTS subpackage (internal): application services import and call `TTSEngine` for GPU inference, model management, and chapter parsing (`DEC-tts-as-backend-module`)
- Model adapter layer (internal): per-model loading and inference is delegated to concrete adapters implementing the `ModelAdapter` protocol (`src/local_tts/tts/adapters/`). New models are enabled by adding an adapter and registering it in `_ADAPTER_REGISTRY`; `ModelLoader.list_models()` exposes a `loader_available` flag (surfaced through `GET /models`) so the frontend can hide download/load actions for models without an adapter. **License metadata:** each `COMPATIBLE_MODELS` entry is a `ModelCatalogEntry` carrying `name`, `license`, `license_is_foss`, and an optional `license_notice` (verified against the model's HuggingFace card — the **model-weights** license governs, so a model with FOSS code but non-commercial weights such as XTTS-v2/fish-speech/F5-TTS is `license_is_foss=false`); these three fields are surfaced on `ModelInfo` and the `GET /models` response. Every non-FOSS model (`license_is_foss=false`) must carry a non-empty `license_notice` and FOSS models must carry none, so the frontend can disclose usage terms (`DEC-model-license-disclosure`). **Language convention:** the application layer passes the canonical **ISO 639-1 code** (`it`, `en`, … — same as preprocessing / `DEC-default-italian-language`) to `synthesize(language=…)`; each adapter must translate that code to its model-specific identifier (e.g. Kokoro `it`→`i` via `_resolve_lang_code`; the Qwen3 adapter `it`→`Italian` via `_resolve_language`), still accept its native identifiers for backward compatibility, and raise a clear `ValueError` for an unrecognized language. A model that **auto-detects** the language from the text (no language tag) is a recognized variant: its adapter still accepts the ISO code, validates it (raising `ValueError` on a malformed value) and defaults it to Italian, but the value is **advisory** — it is not forwarded to the model (VoxCPM2, MOSS-TTSD and Fish S2-Pro `_resolve_language`). Such an adapter should also disable any built-in text normalizer the model exposes, since the app already feeds the exact user-confirmed, preprocessed text (`DEC-preprocess-review-flow`) — VoxCPM2 and Fish S2-Pro pass `normalize=False` (`ServeTTSRequest.normalize` defaults `True`). A **dialogue / multi-speaker** model used for single-voice narration is a further recognized variant: MOSS-TTSD (`OpenMOSS-Team/MOSS-TTSD-v1.0`, `AutoModel`/`AutoProcessor` + `trust_remote_code` + companion `OpenMOSS-Team/MOSS-Audio-Tokenizer`) wraps each input in a single `[S1]` speaker turn and generates with no reference clip by default (an optional `voice` reference clip + `prompt_text` transcript enables zero-shot cloning) — a deliberate architectural-mismatch accommodation, not a bug. Its inference path mirrors the model's published `AutoProcessor`/`AutoModel` workflow in **generation** mode — a single `user` message (the processor requires generation mode to be an odd-length conversation ending in `user`; `continuation` mode needs a trailing `assistant` turn and would raise), with the optional reference clip attached to the user message's `reference` (the processor encodes the path itself). It is verified by mocked unit tests; runtime validation against the ~19 GB weights is a GPU-host step. **`trust_remote_code` version hardening (convention for any remote-code adapter):** (a) **prefer pinning the model repo `revision`** in `from_pretrained` so the downloaded Python cannot silently drift to HEAD — **but not when the loader forwards `revision` into a different-repo sub-component**: MOSS-TTSD's `AutoProcessor.from_pretrained` forwards every extra kwarg (incl. `revision`) into the companion codec's `AutoModel.from_pretrained` (`OpenMOSS-Team/MOSS-Audio-Tokenizer`, a separate repo with unrelated commit hashes), so a model-revision pin 404s the codec; MOSS-TTSD therefore loads at HEAD and relies on the transformers baseline for stability (`_MODEL_REVISION` is retained as a documented reference only, not passed); (b) **bridge renamed transformers symbols** before loading — MOSS-TTSD's companion audio tokenizer (`OpenMOSS-Team/MOSS-Audio-Tokenizer`, loaded via `codec_path`) imports the transformers 5.x name `PreTrainedConfig`, so `_install_transformers_compat()` idempotently aliases it onto `PretrainedConfig` (same class, renamed in 5.x) — a **defensive no-op on the project's transformers 5.x baseline** (`DEC-transformers-5x-baseline`), retained to fail gracefully if ever run on an older transformers. MOSS-TTSD and its codec genuinely require **transformers ≥5.0** (the model's remote code also imports the 5.x `transformers.initialization` module), which drove the backend baseline to transformers ≥5.x (`DEC-transformers-5x-baseline`); that baseline is now **exploratory** and the installed major is flipped per model under trial (currently 4.57.3, to support Qwen3-TTS / Fish S2-Pro — under which MOSS-TTSD is not loadable at runtime). **Fish Audio S2-Pro** (`fishaudio/s2-pro`, `FishS2ProAdapter`) is a further auto-detect variant: it loads via the **`fish-speech` v2.x engine** (`TTSInferenceEngine` built from `launch_thread_safe_queue` + `load_decoder_model`, driven by `ServeTTSRequest`/`ServeReferenceAudio`; 44.1 kHz read from the DAC decoder; optional reference-clip zero-shot cloning) and is **non-FOSS** (Fish Audio Research License → `license_is_foss=false` + notice). Its `fish-speech` package is **GitHub-only** and hard-pins `torch==2.8.0` / `transformers<=4.57.3` — the **inverse** of MOSS-TTSD/Higgs v3 — so it is a **GPU-host dependency**: the adapter lazy-imports it (raising a clear install hint), tests mock it, and full-weight validation is a GPU-host step. This drove the baseline into an **exploratory** state (pin widened to `transformers>=4.38`; the operator flips the installed major per model; currently `4.57.3`, under which MOSS-TTSD/Higgs v3 are not loadable at runtime — `DEC-transformers-5x-baseline`). The S2-Pro and **Qwen3-TTS** adapters both stay **registered** as GPU-host-dependency adapters (their packages lazy-import; they are never backend runtime deps): loadability is purely a runtime function of the installed transformers major. **Qwen3-TTS** (`Qwen/Qwen3-TTS-12Hz-1.7B-CustomVoice`, `Qwen3TTSAdapter`, `qwen-tts` package) requires **transformers 4.57.3** and runs **natively** there with no shims (verified end-to-end on GPU: load + 24 kHz Italian generation); at a transformers ≥5.x install it is registered but not loadable (its `qwen-tts` model code breaks on multiple 5.x API drifts — `check_model_inputs`, removed config special-token attrs, the RoPE-init refactor — which the user declined to shim, choosing instead to run the version `qwen-tts` requires). Installed on the GPU host with `pip install qwen-tts` when the baseline is 4.57.3. **XTTS-v2** (`coqui/XTTS-v2`, `XTTSV2Adapter`) is a **translate-ISO** adapter (like Kokoro/Qwen3 — it takes an *explicit* language: `it`→`it`, only `zh`→`zh-cn`, NOT auto-detect) loaded via the Coqui `coqui-tts` **low-level `Xtts` API** from the HF-cached snapshot (`XttsConfig().load_json` → `Xtts.init_from_config` → `Xtts.load_checkpoint(checkpoint_dir=…)`, then `Xtts.synthesize(text, speaker=…, speaker_wav=…, language=…)` returning `{"wav": …}` — coqui-tts ≥0.27 makes those keyword-only and deprecates the old `config` positional and `speaker_id` kwarg). It uses a named built-in cross-lingual studio speaker by default (selected by `voice`, default `Claribel Dervla`) **or** zero-shot cloning from an optional `speaker_wav` reference clip; outputs 24 kHz; and is **non-FOSS** (Coqui Public Model License → `license_is_foss=false` + notice, `DEC-model-license-disclosure`). Unlike VoxCPM2/Fish S2-Pro, XTTS exposes **no flag to disable** its internal tokenizer-level text cleaning, so the `normalize=False` confirmed-text lever does not apply — a documented limitation (the upstream preprocessing already verbalizes numbers/dates, so the residual cleaning is largely a no-op). **Unlike Qwen3-TTS / Fish S2-Pro, the maintained `coqui-tts` fork (0.27.5) is *compatible* with the baseline** (`transformers>=4.57`/no torch pin/`numpy>=1.26`/Python >=3.10,<3.15 — verified 2026-06-21 to install alongside transformers 4.57.3 / torch 2.10 / numpy 2.x without downgrading, and its `config.languages` + 24 kHz output rate match the adapter constants); because it does **not** conflict (unlike qwen-tts / fish-speech) it is a **declared backend dependency** (`coqui-tts>=0.27` in `pyproject.toml`, like kokoro/voxcpm), lazy-imported at load (clear install hint), mocked in tests, and stays **registered** (`loader_available=true`). The *archived original* `TTS` 0.22.0 is the incompatible one (old pins, Python <3.12). Adapter API validated against the real installed package (`Xtts.synthesize`/`load_checkpoint` signatures); full-weight generation remains a GPU-host step. **CosyVoice 3** (`FunAudioLLM/Fun-CosyVoice3-0.5B-2512`, `CosyVoice3Adapter`) is an **auto-detect** variant (like VoxCPM2/MOSS-TTSD/Fish S2-Pro — `_resolve_language` validates the ISO code + defaults Italian but does **not** forward it) loaded via the FunAudioLLM `cosyvoice` package's own `AutoModel` (`from cosyvoice.cli.cosyvoice import AutoModel`, `AutoModel(model_dir=…)` from the HF-cached snapshot — **not** HuggingFace's `AutoModel`); inference via `inference_zero_shot(text, prompt_text, prompt_wav_path)` (when a `prompt_text` transcript is given) or `inference_cross_lingual(text, prompt_wav_path)`, each yielding a generator of `{"tts_speech": <tensor>}` chunks that the adapter concatenates; 24 kHz (read from `model.sample_rate`); Apache-2.0 (FOSS). **Notable variation — a reference clip is mandatory:** CosyVoice 3 has **no built-in/SFT speaker** (the snapshot ships no `spk2info.pt` and no bundled prompt), so `synthesize` **requires** a `voice` reference-clip path (the first adapter with no built-in voice). **Temporary stopgap until Phase 6 voice selection (`TASK-voice-language-selection-ui`):** when no `voice` is supplied the adapter falls back to a user-provided default clip at `config.DEFAULT_VOICE_PATH` (repo-root `wavs/default.mp3`, gitignored) via cross-lingual cloning when that file exists, else raises a clear `ValueError`. The path is read at call time so the clip can be added/removed without reloading. Its `cosyvoice` package **hard-pins `transformers==4.51.3` / `torch==2.3.1` / `numpy==1.26.4`** (a third distinct constraint vs. qwen-tts/fish-speech/MOSS-TTSD), so per `DEC-transformers-5x-baseline` it is a **GPU-host dependency** (lazy-imported with a clear install hint, mocked in tests, **not** in `pyproject.toml`), registered uniformly; full-weight generation is a GPU-host step. **F5-TTS** (`SWivid/F5-TTS`, `F5TTSAdapter`) is an **auto-detect** variant (like CosyVoice 3 / VoxCPM2 — `_resolve_language` validates the ISO code + defaults Italian but does **not** forward it) loaded via the `f5-tts` package's high-level `f5_tts.api.F5TTS` class (`F5TTS(model="F5TTS_v1_Base", device=…)`, checkpoint resolved from the HF-cached `SWivid/F5-TTS` snapshot); inference via `F5TTS.infer(ref_file, ref_text, gen_text)` returning `(wav, sample_rate, spectrogram)` from which the adapter extracts the 1-D wav; 24 kHz (read from `model.target_sample_rate`). Like CosyVoice 3 it has **no built-in speaker** — synthesis is always zero-shot cloning, so `synthesize` **requires** a `voice` reference-clip path (`prompt_text` transcript optional — omitted → empty `ref_text` triggers F5-TTS's internal ASR auto-transcription) and uses the **same Phase-6 stopgap**: when no `voice` is given it falls back to the user-provided default clip at `config.DEFAULT_VOICE_PATH` (repo-root `wavs/default.mp3`, gitignored) when present, else raises a clear `ValueError`. It does no number/abbreviation expansion of its own, so the `normalize=False` lever does not apply. It is **non-FOSS** (CC-BY-NC-4.0 weights → `license_is_foss=false` + notice, already in `COMPATIBLE_MODELS` from Phase 5.2). Its `f5-tts` package pins `numpy<=1.26.4` on Python ≤3.10 (conflicting with the baseline's numpy 2.x) and pulls a heavy dependency tree, so per `DEC-transformers-5x-baseline` it is a **GPU-host dependency** (lazy-imported with a clear install hint, mocked in tests, **not** in `pyproject.toml`), registered uniformly; full-weight generation is a GPU-host step. **Higgs Audio v3** (`bosonai/higgs-audio-v3-tts-4b`, `HiggsAudioV3Adapter`) is **non-FOSS** (Boson Research & Non-Commercial License → `license_is_foss=false` + notice) and is **intentionally UNREGISTERED** — it stays **listed** (`loader_available=false`, with its license notice) like other no-adapter models, but cannot be loaded. Reason (verified 2026-06-21 against a runtime "does not recognize this architecture" failure): Boson publishes v3 **only as a server** (vLLM-Omni / SGLang-Omni; the model card shows no transformers/Python path), its `model_type: higgs_multimodal_qwen3` is **absent from `transformers`** (released 5.12.1 *and* `main`), and the HF repo ships **no remote code** (no `auto_map`/`*.py`), so `trust_remote_code` can't help and there is **no in-process loader** under `DEC-single-process`. A server-backed loader was considered and declined (it would break `DEC-single-process`). The `higgs_audio_v3.py` module + 38 mocked tests are **retained** on the bet that transformers later adds native `higgs_multimodal_qwen3` support, at which point the in-process `text-to-speech` pipeline path (it imports `pipeline` at module top-level so tests can patch `local_tts.tts.adapters.higgs_audio_v3.pipeline` — patching `transformers.pipeline` does **not** reliably intercept `from transformers import pipeline`) can be re-registered. **Lesson for future adapters:** confirm the model is actually published for in-process `transformers` (architecture present in the library *or* remote code in the repo) before treating a model-card `pipeline`/`AutoModel` snippet as runnable — a server-only model (SGLang/vLLM serving) does not satisfy `DEC-single-process`. See `architecture.md` § Model-Specific Loading Requirements.
- Preprocessing service (internal, `src/local_tts/preprocessing/`): a dedicated application service — **not** in the TTS subpackage (`DEC-text-preprocessing-pipeline`). A `Pipeline` runs an ordered list of discrete `Stage` objects (the `Stage` protocol + a name-keyed registry in `stages.py`), configured along two axes — language profile + model profile (`profiles.py`) — with an optional on-disk domain dictionary tolerant of absence. `PreprocessingService.preprocess(text, language, model_id)` returns normalized text + before/after char counts. New stages are added by implementing the `Stage` protocol under the canonical name constants (`STAGE_*`) and registering them via `register_stage` in `preprocessing/__init__.py`; the default model profile runs the registered subset of `DEFAULT_STAGE_ORDER`. Stage convention (established by `unicode_sanitization.py`, the first concrete stage): one module per stage at `preprocessing/<stage_name>.py`; universal tables live in the module while language-specific data is exposed as a module-level `BUILTIN_LANGUAGE_DATA` mapping (`{language: {namespace: {...}}}`, the namespace named after the stage) and registered via `register_language_data` next to `register_stage` in `__init__.py` (a stage with no language-specific data omits `BUILTIN_LANGUAGE_DATA` and just registers the stage, as `layout_repair.py` and `sentence_segmentation.py` do); behavior switches read from the model profile's `params` via `PARAM_*` constants with safe defaults — no language/model rule is hardcoded in the shared `run` logic (`DEC-text-preprocessing-pipeline`). The default order is: unicode sanitization → layout repair → numeric/symbolic verbalization → abbreviation expansion → **sentence segmentation**. `sentence_segmentation.py` (the fifth stage) puts each sentence on its own line; the synthesizer chunks the confirmed text **one line per segment** (`split_into_sentences` in `tts/synthesizer.py` splits on newlines only, not punctuation), so this stage *defines* the TTS chunking and the reviewed text matches the spoken chunks. It runs **last** because splitting on a sentence-ending period must happen after numbers/abbreviations are expanded (otherwise `11.988` / `E.F.` are mis-split) — the inverse of layout repair, which *reflows* soft-wrapped fragments and keeps heading/list/**bare-number** lines standalone so structural lines are not glued into prose. It also performs **dialogue isolation**: a spoken span delimited by directional guillemets (`«` … `»`) is put on its own line(s) by breaking before `«` and after `»` (so narration / quote / trailing dialogue tag are distinct chunks), then flattens `«`/`»` to `"`. For this to work the **Unicode stage deliberately does NOT flatten `«`/`»`** (it flattens every other quote variant) — the directional guillemets carry chunking information consumed downstream; any new quote-related handling must preserve that contract. A stage may delegate language-specific work to a declared library selected by the language profile rather than branch on the language itself: `numeric_symbolic_verbalization.py` (the third concrete stage) spells numbers/ordinals/years via `num2words` (LGPL, a direct dependency) keyed by a `num2words_lang` code in its `BUILTIN_LANGUAGE_DATA`, with every other table (separators, month names, currency/percent/symbol words) in that same language data. A requested output language with **no** registered data is rejected by `resolve_language_profile` with `UnsupportedLanguageError` (surfaced by `/preprocess` as a 400) rather than degrading to a no-op — preprocessing rewrites are language-specific, so a passthrough would misrepresent the raw text as "normalized"; an empty/omitted language falls back to the supported default (`it`). Individual stages remain defensively no-op when the resolved (supported) language data lacks their own namespace. The `model_id` of the loaded model is supplied by the caller (the `/preprocess` API layer reads it from the Model Service and owns the "no model loaded" error), keeping the service unit-testable without a GPU.
- Preprocess & synthesis API (`src/local_tts/api/preprocess.py`, `jobs.py`): the synchronous `POST /preprocess` accepts a multipart `.txt` **file** _or_ raw `text` plus an optional `language`, runs the Preprocessing Service with the **currently loaded model's** profile, and returns the normalized text with before/after char counts for review (`REQ-USA-normalized-text-review`). The API layer owns input validation (`.txt`/≤2 MB/UTF-8/non-empty, exactly-one-of file/text, and an unsupported output `language` → 400) and the "no model loaded" error (409), reading `tts_engine.loaded_model_id` so the service stays GPU-agnostic; `REQ-F-upload-text-file` is satisfied here. Language validation precedes the model check (input-validation-before-model-check), so an unsupported language returns 400 even with no model loaded. `POST /jobs/synthesis` then takes JSON `{text, source_filename, voice?, language?}` and synthesizes **exactly** that confirmed text with **no** re-preprocessing (disk preflight estimated from `len(text)`). This two-call, transient-normalized-text contract is the preprocess-then-confirm flow (`DEC-preprocess-review-flow`). The `PreprocessingService` is instantiated once in the app lifespan and attached to `app.state`. All routes mount under `/api/v1`.
- Library & playback API (`src/local_tts/api/audiobooks.py`, `playback.py`): the `LibraryService` backs audiobook list/get/delete (delete cascades to chapter records and on-disk audio) and resolves chapter audio paths; chapter audio is streamed inline via `FileResponse` with HTTP Range support (206/`Content-Range`/416). Playback resume is a two-level bookmark (audiobook-level last chapter + per-chapter timestamp) served by the dedicated `PlaybackService` under `GET`/`PUT /audiobooks/{id}/position`.
